# 自动化暴露面探测能力评估（基于当前代码）

## 结论（TL;DR）

当前仓库已经具备**“授权实验室环境下的半自动暴露面探测原型”**能力，尤其是：

- 目标资产输入（域名/IP/CIDR）到可探测目标集合的展开；
- 受策略约束（allowlist/open）的主动探测执行；
- 多协议观测（DNS / TCP / UDP spike / SCTP INIT / HTTPS / 部分 SBI HTTP/2 路径）；
- 探测结果到暴露面候选与风险假设的结构化输出；
- Agent/Skill 编排与图谱/GraphRAG 关联分析。

但如果你的目标是**“可直接用于生产级、持续化、自动化暴露面管理（ASM/EASM）”**，目前还不够，缺少关键能力：

1. 持续资产发现（被动+外部情报）与资产生命周期管理；
2. 探测深度和插件化能力（协议覆盖、误报控制、漏洞验证链）；
3. 调度/队列/重试/分布式执行器；
4. 结果归一化、去重聚类与长期趋势分析；
5. 企业级安全治理（RBAC、审计、隔离、合规边界）；
6. 产线级可观测性（SLO、指标、告警、回归基线）。

## 当前“能做到什么”

### 1) Outside-in 输入到探测目标生成

- 支持 domains / ips / cidrs / extra_hosts 合并展开、去重，并限制 CIDR 展开规模（避免一次性失控）。
- 这说明你已经有了“外部资产集合 → 可执行探测集”的入口层。

### 2) 探测执行（实验室授权场景）

- Probe 有清晰策略门控：`probe_enabled`、`probe_mode=allowlist|open`、suffix allowlist、CIDR allowlist。
- 单目标会执行多种探测：
  - DNS 解析；
  - TCP 端口连通性；
  - UDP 常见端口与 spike（含 IKE/GTP-U 方向）；
  - SCTP INIT（依赖 Scapy，缺失时会显式报告）；
  - HTTPS HEAD 与证书摘要；
  - SBI 未授权路径探测（HTTP/2 优先）。

### 3) 结果结构化与暴露候选生成

- Probe 输出是结构化模型（每目标含开放端口、service_hints、banner、SBI 结果等）。
- Exposure 服务可将 probe 结果转成候选暴露条目（candidate_fqdn/protocol_stack/risk_hypotheses/confidence）。
- 这意味着你已经完成了“探测事实 → 风险候选”的最小闭环。

### 4) 编排与分析增强

- 后端公开了 `/api/probe/run`、`/api/exposure/generate` 等 API，可供前端与自动流程编排调用。
- 还有 GraphRAG/Agent/Skills 组件，具备把“探测事实 + 知识图谱 + LLM 推理”串起来的框架能力。

## 是否可用于“自动化暴露面探测”？

### 可以，但仅限“原型/实验室级自动化”

如果你定义的“自动化”是：

- 输入资产列表；
- 在授权范围内执行多协议探测；
- 自动产出候选暴露与初步风险假设；

那当前系统**可以**。

### 不足以直接作为“生产级 ASM 平台”

如果你定义的“自动化暴露面探测”是企业产线要求（7x24、全量资产持续发现、低误报、可审计、可追责、可回归），当前实现**还差一大段工程化能力**。

## 还差在哪里（按优先级）

## P0（必须补齐，才能进入真实持续运行）

1. **资产发现链路**
   - 目前更偏“你给我目标，我去探测”；缺少持续发现（CT 日志、DNS 被动数据、云资产 API、WAF/CDN 资产、证书反查）。
   - 需要资产指纹归一化（同资产多 IP/多域映射）与生命周期状态（新增、消失、漂移）。

2. **任务调度与执行框架**
   - 需要定时任务、队列、重试、并发配额、分布式 worker（避免单实例扫描瓶颈）。
   - 需要失败分类（网络故障/策略拒绝/目标不可达/插件异常）与重跑策略。

3. **结果存储与去重基线**
   - 除图谱外，还要面向时序分析的数据模型（按扫描批次、资产、端口、协议、证据做版本化）。
   - 做“变更检测”：新开端口、新暴露路径、认证状态变化（401→200）这类高价值告警。

4. **权限与审计**
   - 谁可以扫哪些资产、何时扫、扫了什么，需要强审计轨迹。
   - 需要 RBAC、操作留痕、审批流（尤其 open mode 的开启关闭）。

## P1（显著提升质量，降低误报漏报）

1. **协议插件体系**
   - 把当前探测逻辑抽成插件接口（discover/probe/validate/report），便于快速扩展 SIP/DIAMETER/GTP-C/5GC SBA 细分检查。

2. **误报控制与证据评分**
   - 对 `service_hints`、banner、SBI 响应建立置信度与冲突消解规则。
   - 引入二次验证（例如多次采样、一致性阈值、跨探针交叉校验）。

3. **漏洞验证边界**
   - 目前更偏“暴露面和可达性”而非漏洞确认。
   - 若要落到漏洞级，需要 PoC 沙箱策略、只读验证模板、强防误伤安全阈值。

4. **可观测性**
   - 增加 Prometheus 指标（任务时延、成功率、协议命中率、超时率、误报回归率）。
   - 增加告警规则与可视化 dashboard（资产风险热力、变化趋势）。

## P2（平台化与业务集成）

1. **工单闭环**：暴露发现 → 指派 → 修复验证 → 关闭；
2. **多租户隔离**：不同项目/客户/实验网隔离策略与数据；
3. **外部系统联动**：SIEM/SOAR/CMDB/工单系统；
4. **规则与模型治理**：提示词版本、规则版本、回归基准集、漂移检测。

## 给你的实操建议（最短路径）

1. 先把当前系统定位为：**“授权实验网暴露面发现引擎 v0”**；
2. 先补 P0 四件套：资产发现、调度队列、时序存储、审计权限；
3. 再做 P1 插件化与误报控制；
4. 最后把 GraphRAG/Agent 放到“解释与辅助决策层”，避免直接驱动高风险主动动作。

---

> 合规提醒：任何主动探测都应在明确授权范围内执行，并保留可审计证据链。
