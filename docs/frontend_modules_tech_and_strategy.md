# 3GPP Open Exposure Demo 项目总结与前端模块技术策略

## 1. 项目整体总结（面向汇报）

`3gpp-open-exposure-demo` 是一个面向实验室场景的前后端分离原型系统，目标是把“标准文本理解”变成“可计算、可追溯、可演示”的工程闭环：

1. 标准文本 -> 结构化抽取（Evidence Pack + Worker/Judge/Repair）。
2. 抽取结果 -> staging graph（人工审核后合并主图）。
3. 主图知识 -> 候选暴露面生成（FQDN/协议/网元/风险）。
4. 候选结果 -> 授权探测与分析报告（DNS/HTTPS/路径推演）。
5. 平台层支持 Agent/Skill 编排与 GraphRAG 混合检索问答。

核心价值不是“做一次问答”，而是形成持续演进的知识工程与验证平台。

---

## 2. 前端技术栈与总体架构

## 2.1 技术栈

- 框架：Vue 3（Composition API）
- 构建：Vite
- 语言：TypeScript
- 状态管理：Pinia
- UI 组件：Element Plus（深色主题）
- 图可视化：Cytoscape.js
- 图表：ECharts
- HTTP：Axios（统一 `client`）

## 2.2 前端结构

- 壳层：`AppShell` 统一导航、头部状态带、页面容器。
- 路由：按业务域划分页面（Dashboard / Graph / Extract / Agent / GraphRAG / Exposure / Experiments / Reports）。
- Store 层：按核心域拆分（图谱、抽取、GraphRAG）。
- API 层：轻量封装，页面通过 store 调用，避免页面直接耦合接口细节。

## 3. 前端模块拆解（技术 + 策略）

## 3.1 Dashboard（总览）

**技术实现**
- 并发加载多源数据：图谱统计、图谱校验、系统状态、实验任务、Agent 运行、最近探测、抽取状态。
- 使用 ECharts 渲染节点/关系分布柱状图。

**策略**
- 把“运行可用性”前置：Neo4j、LLM、Agent、Probe 状态统一在首页暴露。
- 强调可观测性：不仅看业务结果，还看系统健康与最近运行轨迹。

## 3.2 Graph（知识图谱）

**技术实现**
- Cytoscape 渲染图；支持布局切换、类型过滤、关系过滤、搜索高亮、PNG/JSON 导出。
- 支持子图加载（关键词 + `seed_limit` + `max_edges`）和点击节点邻域扩展。

**策略**
- 性能优先：默认子图而非全量图；大图自动降级布局动画。
- 交互可控：过滤和扩展都在同一页面完成，降低“跳页上下文丢失”。
- 可解释：右侧节点详情展示描述、证据来源、相关边，增强审阅效率。

## 3.3 Extract（标准抽取工作台）

**技术实现**
- 覆盖 run 启动、repair、detail/trace/report 拉取、staging diff、merge 入主图。
- 通过 `extractionStore` 做 run 生命周期管理。

**策略**
- Human-in-the-loop：前端刻意保留“staging 审核 -> merge”步骤，不允许黑盒直接入主图。
- 结果分层展示：Evidence、Workers/Judge、Staging、Trace/Report 分 tab，便于定位问题。

## 3.4 Exposure（候选暴露面）

**技术实现**
- 参数化生成候选（service + MCC/MNC）。
- 支持授权探测（Probe）和暴露面综合分析（含可选 LLM 解释）。
- 支持报告拉取和 CSV 导出。

**策略**
- 研究边界清晰：UI 文案与流程强调“授权环境探测”，防止误解为公网扫描工具。
- 一体化验证链：候选、探测、风险、路径、报告在一个页面闭环完成。

## 3.5 Agent / Skill

**技术实现**
- 展示技能清单（可发现工具）。
- 支持手动运行单个 skill。
- 支持输入 goal 一键触发 Agent 多步调度并回放 trace。

**策略**
- 平台化导向：把“能力”以 skill 形式结构化，不把逻辑散落在页面按钮中。
- 可追踪导向：每一步输入/输出可见，避免“Agent 黑盒”。

## 3.6 GraphRAG

**技术实现**
- 支持问答流式输出（`query-stream`）。
- 展示回答与引用；默认提供“协议关系 + 标准文档引用”的示例问题。

**策略**
- 用“结构化问答”证明差异：重点让问题落在多跳关系与文档证据，不做纯闲聊问答。
- 体验优先：流式返回减少等待焦虑，引用标签增强可信度与复核性。

## 3.7 Reports（报告导出）

**技术实现**
- 在线渲染 Markdown（校验报告、摘要）。
- 下载 Mermaid、CSV、Markdown 等成果文件。

**策略**
- 交付导向：页面本质是“演示成果出口”，直接服务汇报与留档。

## 3.8 Experiments（实验验证）

**技术实现**
- 展示 mock 实验任务卡片与状态。

**策略**
- 安全合规提示前置：强化“合法授权场景”边界。
- 为后续“任务编排自动化”预留 UI 位。

## 4. 前端工程策略（跨模块）

## 4.1 分层策略

- 页面层：负责交互和视图，不承载复杂业务逻辑。
- Store 层：负责状态聚合、跨 API 流程编排。
- API 层：负责通信细节和流式协议解析。
- 类型层：统一 TS 类型约束，减少跨页字段漂移。

## 4.2 可维护策略

- 统一暗色视觉体系与玻璃态组件风格，保证演示一致性。
- 关键域（Graph/Extraction/GraphRAG）各有独立 store，降低耦合。
- 用显式错误提示与空态组件，降低调试成本。

## 4.3 性能策略

- 图谱默认走子图检索，避免大图首屏卡顿。
- 复杂布局按规模自动关闭动画。
- 路由组件懒加载，减小首包压力。

## 4.4 可信与可解释策略

- 抽取展示 Evidence/Judge/Conflict/Trace，全链路可追踪。
- GraphRAG 与 Exposure 都保留 citations/evidence 语义，避免只给结论。
- 报告页面提供结构化导出，支撑复盘与复现。

## 5. 当前前端定位与后续建议

## 5.1 当前定位

当前前端已经不只是“可视化壳子”，而是承担了：
- 运行态观测面板
- 人工审核入口（staging merge）
- 分析与报告交付工作台

## 5.2 下一步建议（前端）

1. 新增统一“诊断模式”抽屉：显示每次请求的检索命中、耗时、错误分类。
2. 把 GraphRAG 与 Graph 页联动增强：点击引用可直接定位图中实体。
3. 增加结果对比组件：GraphRAG vs 普通 RAG 的 A/B 输出同屏对照。
4. 增加 E2E 回归脚本（关键页面最小演示路径），保证演示稳定。

---

## 6. 一句话结论（可直接用于答辩）

本项目前端以“可观测、可审核、可解释、可交付”为核心策略，围绕图谱构建、抽取审核、暴露分析与 GraphRAG 问答形成统一工作台，使研究型能力具备稳定演示与持续迭代的工程基础。
